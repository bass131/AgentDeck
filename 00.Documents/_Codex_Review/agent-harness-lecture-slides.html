Part 0 · Opening

Agent Harness는AI 코딩을 위한안전한 작업장입니다

AI가 코드를 “그냥 고치는” 것이 아니라, 문서로 방향을 잡고, 역할을 나누고, 테스트로 확인하고, 위험한 순간에는 사람에게 멈춰 묻도록 만드는 운영 시스템입니다.

Part 1 · 목표

오늘은 코드를 몰라도 이해할 수 있게 설명합니다

1

왜 필요한가?

AI가 빨리 코딩할수록 실수도 빨리 퍼질 수 있습니다. Harness는 속도와 안전을 같이 잡기 위한 장치입니다.

2

어떤 부품이 있나?

문서, 헌법, 정책, Phase, SubAgent, Hook, 테스트 게이트가 서로 맞물립니다.

3

작업은 어떻게 흐르나?

큰 목표를 작은 Phase로 쪼개고, 담당 에이전트가 실행하고, 기계와 사람이 각각 판단합니다.

Part 1 · 비유

AgentDeck을 작은 공장이라고 생각하면 쉽습니다

AI는 아주 빠른 작업자입니다. 하지만 작업자가 많아질수록 “누가 무엇을 고쳤는지”, “완료 기준은 무엇인지”, “위험한 버튼은 누르면 안 되는지”가 중요해집니다.

설계도

00.Documents가 제품 방향과 아키텍처를 설명합니다.

공장 규칙

CLAUDE.md가 AI가 절대 어기면 안 되는 헌법입니다.

작업표

01.Phases가 큰 일을 작은 작업 단위로 쪼갭니다.

검사대

typecheck, test, lint, reviewer가 결과를 검사합니다.

Part 2 · 문제 정의

AI 코딩의 문제는 “실력”보다 “통제”입니다

Harness가 없을 때

  • AI가 프로젝트 규칙을 잊고 다른 방식으로 구현할 수 있습니다.
  • 작업 범위가 조금씩 새어 나가서 의도와 다른 파일을 고칠 수 있습니다.
  • 테스트 없이 “완료했습니다”라고 말할 수 있습니다.
  • push, 배포, 설정 변경처럼 위험한 행동을 너무 쉽게 할 수 있습니다.

Harness가 있을 때

  • AI는 먼저 문서와 규칙을 읽고 움직입니다.
  • 작업은 Phase 단위로 쪼개져 완료 조건이 명확해집니다.
  • 도메인별 SubAgent가 자기 영역만 맡습니다.
  • 기계가 판정할 수 있는 일은 테스트로 확인하고, 위험한 일은 사람에게 멈춰 묻습니다.

Step 01

사용자 목표

사람은 큰 방향을 줍니다. AI는 아직 코드를 만지지 않고, 먼저 무엇을 해야 하는지 이해합니다.

AgentDeck에서 실제로 대응되는 파일 CLAUDE.md, 00.Documents/PRD.md
왜 필요한가? 목표를 제대로 이해하지 못한 상태에서 코드를 고치면, 빠른 속도로 엉뚱한 방향으로 갈 수 있습니다.
이 단계의 완료 신호 목표, 범위, 멈춰야 할 위험 조건이 말로 설명됩니다.

Part 3 · Layer 1

00.Documents는 AI가 보는 제품 설명서입니다

PRD.md 무엇을 만들지 설명합니다. 제품 요구사항과 제외할 범위를 정합니다.
ARCHITECTURE.md 어떻게 만들지 설명합니다. 폴더 구조, 데이터 흐름, 신뢰 경계를 잡습니다.
ADR.md 왜 이런 결정을 했는지 기록합니다. 나중에 같은 논쟁을 반복하지 않게 합니다.
UI.md 화면이 어떤 원칙으로 보여야 하는지 정합니다. 색, 밀도, 레이아웃, 안티패턴을 담습니다.
FEATURE_MAP.md 구현된 기능과 남은 기능을 추적합니다. “우리가 어디까지 왔는지”를 보여줍니다.

Part 3 · Layer 2

CLAUDE.md는 AI 작업자의 헌법입니다

헌법이 하는 일

  • AI가 매번 떠올려야 할 절대 규칙을 둡니다.
  • 프로젝트 문서 지도를 알려줍니다.
  • 위험한 행동과 사람 확인이 필요한 순간을 정의합니다.
  • 비전공자 사용자를 위한 설명 방식도 지정합니다.

특히 중요한 규칙

  • 파일 시스템, 프로세스 실행, DB 접근은 Electron main 프로세스만 합니다.
  • 코딩 엔진은 반드시 AgentBackend를 통해 부릅니다.
  • API 키와 시크릿은 코드에 저장하지 않습니다.
  • 새 기능은 테스트 먼저 작성하는 방식이 기본입니다.

Part 3 · Layer 3

.claude 폴더는 하네스의 조종실입니다

agents

main-process, renderer, qa처럼 역할별 SubAgent 설명서가 들어 있습니다.

policies

작업 등급, 위험 깃발, 리뷰 기준, 루프 운영 방식을 정합니다.

skills

큰 목표를 Phase로 쪼개는 work-plan과 실행하는 work-run이 있습니다.

commands

/harness, /review, /session:start 같은 명령 흐름을 담습니다.

templates

Phase 문서와 완료 보고서의 기본 양식을 제공합니다.

hooks

위험 명령, 테스트 누락, 규칙 위반을 자동으로 감지하는 안전장치입니다.

Part 4 · Phase

큰 목표는 바로 구현하지 않고 Phase로 쪼갭니다

큰 목표 그대로 시키면

“배포 시스템 완성해줘”처럼 큰 요청은 범위가 넓어서 AI가 중요한 순서를 놓치거나, 여러 파일을 한 번에 흔들 수 있습니다.

Phase로 쪼개면

“1단계: 현재 배포 갭 확인”, “2단계: 패키징 설정”, “3단계: 업데이트 검사”처럼 끝나는 지점과 검증 기준이 생깁니다.

Part 4 · Phase Anatomy

좋은 Phase 문서는 AI에게 작업 지시서가 됩니다

목표이번 Phase가 끝나면 무엇이 달라지는지 한 문장으로 씁니다.
사전 조건시작 전에 완료되어야 하는 앞 Phase나 필요한 문서를 적습니다.
작업 내용해야 할 일을 체크리스트로 나눕니다.
완료 조건typecheck, test, lint, build처럼 측정 가능한 기준을 적습니다.
담당 도메인main-process, renderer, shared-ipc, qa 등 어느 영역인지 정합니다.
위험 깃발trust-boundary, ui-visual, irreversible 같은 정지 조건을 표시합니다.

Part 5 · SubAgent

SubAgent는 역할이 정해진 전문 작업자입니다

SubAgent 담당 영역 비전공자식 설명
main-process Electron main, 파일, Git, LSP, IPC handler 앱의 엔진룸을 맡는 작업자입니다.
agent-backend Claude/Codex 어댑터, AgentEvent 정규화 AI 엔진과 앱 사이의 통역사입니다.
renderer React UI, 컴포넌트, 테마, 상태 사용자가 보는 화면을 만드는 디자이너 겸 프론트 개발자입니다.
shared-ipc main과 renderer 사이의 타입 계약 부서 사이에 주고받는 공식 양식을 관리합니다.
qa 테스트, 회귀 검증, 픽스처 품질 검사 담당자입니다.
reviewer 코드 변경 리뷰 완성품을 다시 확인하는 검사관입니다.

Part 5 · Routing

작업 크기에 따라 움직이는 방식이 달라집니다

단순

한 파일, 작은 수정. 메인 세션에서 직접 처리합니다.

보통

한 도메인 중심. Worker SubAgent 1명이 맡습니다.

복잡

여러 단계가 필요합니다. coordinator가 나누고 reviewer가 봅니다.

대규모

여러 Worker, plan-auditor, reviewer가 필요합니다.

Part 6 · Work Judge

무엇은 AI에게 맡기고, 무엇은 사람이 봐야 할까요?

A

기계 판정

typecheck, test, lint, build처럼 통과/실패가 명확한 일입니다. AI가 자율로 진행합니다.

B

육안/취향

UI 색, 여백, 보기 좋은지 같은 일입니다. 기능은 진행하되 최종 판단은 사람이 봅니다.

C

판단/비가역

push, PR, merge, 배포, 보안 경계 변경처럼 되돌리기 어려운 일입니다. 반드시 사람에게 멈춰 묻습니다.

Part 6 · Loop

loop-driven은 “매번 물어보지 않고, 게이트에서만 멈추는 방식”입니다

예전 방식

AI가 한 단계 할 때마다 “다음도 할까요?”라고 묻습니다. 안전해 보이지만 작업 속도가 크게 떨어집니다.

하네스 방식

처음에 방향과 완료 기준을 정합니다. 그 다음에는 테스트나 리뷰 같은 게이트까지 AI가 스스로 진행합니다. 위험한 순간에만 멈춥니다.

Part 6 · work-run

work-run은 Phase를 실제 코드 변경으로 바꾸는 실행 루프입니다

스캔pending Phase 찾기
정렬의존성 순서 계산
위임도메인 Worker 배정
수정Worker가 파일 변경
리뷰위험 깃발이면 reviewer
검증typecheck/test/lint/build
기록commit, pin, 보고

중요한 점은 Worker가 마음대로 commit하거나 배포하지 않는다는 것입니다. commit과 위험 작업은 별도 게이트를 거칩니다.

Part 7 · Hooks

Hook은 AI 작업 중 자동으로 울리는 경보 장치입니다

예시

  • 위험한 명령을 감지합니다.
  • 테스트 없이 기능을 구현하려 하면 경고합니다.
  • 위험 깃발이 필요한 파일 변경을 감지합니다.
  • 작업 범위나 파일 크기가 너무 커지는지 봅니다.

비유

Hook은 작업자 뒤에서 지켜보는 안전 센서입니다. 모든 판단을 대신하지는 않지만, 위험한 패턴을 보면 “잠깐, 이건 확인해야 합니다”라고 알려줍니다.

Part 7 · Trust Boundary

AgentDeck에서 가장 중요한 보안 원칙은 신뢰 경계입니다

renderer

사용자가 보는 화면입니다. 브라우저와 비슷하므로 직접 파일을 만지면 안 됩니다.

preload

허가된 통로만 열어주는 창구입니다. 필요한 IPC만 화이트리스트로 노출합니다.

main

파일, 프로세스, Git, DB 같은 권한 작업을 처리하는 안전 구역입니다.

Part 8 · Backend Adapter

AI 엔진은 AgentBackend라는 통일된 어댑터 뒤에 숨깁니다

UI사용자 입력
IPCmain으로 요청
AgentBackend공통 인터페이스
Claude/Codex엔진별 구현
AgentEvent공통 이벤트로 정규화

비전공자식으로 말하면, AgentBackend는 여러 외국어를 같은 한국어 양식으로 번역하는 통역 창구입니다. 그래서 UI는 Claude인지 Codex인지 자세히 몰라도 됩니다.

Part 9 · Done Judge

완료 판정은 사람이 느낌으로 하지 않고 기계가 합니다

기계 판정 게이트

  • npm run typecheck: 타입이 맞는지 확인합니다.
  • npm run test: 기존 기능이 깨지지 않았는지 확인합니다.
  • npm run lint: 코드 규칙을 지켰는지 확인합니다.
  • npm run build: 실제 배포용 빌드가 되는지 확인합니다.

왜 중요한가?

AI가 “완료했습니다”라고 말하는 것보다, 테스트 출력이 남아 있는 것이 훨씬 믿을 만합니다. 하네스는 이 출력이 대화 기록에 남도록 요구합니다.

Part 11 · Human Role

사람은 모든 줄을 붙잡는 사람이 아니라 방향과 판단을 맡습니다

AI가 잘하는 일

  • 반복 작업
  • 테스트 작성과 실행
  • 파일 구조 탐색
  • 정해진 규칙 안에서 구현

사람이 해야 하는 일

  • 제품 방향 결정
  • 설계 분기 선택
  • 배포, PR, merge 같은 비가역 행동 승인
  • UI 미감과 사용성 판단
