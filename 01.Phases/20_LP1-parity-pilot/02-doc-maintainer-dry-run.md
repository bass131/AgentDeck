# Phase 02: Doc Maintainer 루프 손실행 — 문서 인용 실측 대조 보고서

## 🎯 목표

P01 목록의 문서가 인용하는 경로·이름·수치·명령이 실제 레포와 일치하는지 대조하고, 불일치는 **고칠 제안만** 보고서로 낸다. 수정 실행 없음 — Doc Maintainer 루프의 1회 손실행이며, 실패 비용은 보고서 한 장이다.

## ⏪ 사전 조건

- P01 완료 (`LP1-01-대상선정.md`의 대상 목록·대조 범위 확정).

## 📝 작업 내용

1. 각 대상 문서에서 검증 가능한 인용을 추출 — 유형: 파일 경로 / 심볼·채널명 / 수치(개수·버전) / 명령어 / 상태 서술("구현 완료" 등).
2. 인용마다 레포 실측 (Read·Grep·Glob — 읽기 전용). 대조 표 스키마:

   | 문서 좌표(파일:줄) | 문서 주장 | 실측 결과(근거 좌표) | 판정 | 제안(불일치 시) |

   판정 값은 **일치 / 불일치 / 미확인** 3종만. "미확인"은 사유 명기 — 억지로 판정하지 않는다.
3. 요약 절: 문서별 일치율 + 불일치 목록 + 제안 목록.
4. 대조 범위는 P01 문서에 적힌 대로 — 실행 중 임의 확장·축소 금지. 범위 조정이 필요하면 보고서에 사유를 기록하고 원 범위 기준 결과를 함께 남긴다.
5. 산출물: `00.Documents/reports/LP1-02-doc-maintainer-손실행.md`

## ✅ 완료 조건 (정량)

- [ ] 보고서 존재, P01 목록의 전 문서가 대조 표에 등장.
- [ ] 모든 행에 문서 좌표(파일:줄)와 판정 존재 — 판정 값은 일치/불일치/미확인 3종만.
- [ ] 불일치 행 전부에 실측 근거 좌표 + 제안 1줄.
- [ ] 기존 문서 diff 0 (`git status` — 신규 보고서 1개 외 00.Documents 변경 없음).

## 📚 학습 포인트

- 문서 부패(doc rot — 문서가 코드 변화를 따라가지 못해 거짓이 되는 현상)를 기계 절차로 잡는 법: "주장 vs 실측" 분리.
- "미확인"을 억지로 판정하지 않는 규율 — 계약 철칙 ①(증거 없는 ✅ 금지)의 문서판.

## ⚠️ 함정

- 발견한 불일치를 그 자리에서 고치고 싶어진다 — **금지**. 이 트랙의 계약은 "제안만". 수정 실행은 영호 채택 후 별도 사이클.
- 실측 근거는 좌표(파일:줄)까지 — "확인했음" 서술만으로는 완료 조건 미달.
