현재 완성도 판단

문서와 소스 구조를 기준으로 본 AgentDeck의 현재 위치다. 테스트 결과는 기존 문서에 기록된 값이며, 이 HTML 생성 과정에서 새로 실행하지는 않았다.

01

권장 아키텍처

기존 `AgentBackend` 계약을 유지하되, Codex를 Claude의 복제 어댑터로 만들지 말고 provider별 capability와 session 의미를 분리한다.

02
Renderer 선택 provider, 모델, mode, message를 명시적으로 전달
IPC Boundary backendId 검증, workspaceRoot 검증, renderer 권한 제한 유지
Registry ClaudeCodeBackend 또는 CodexAppServerBackend 선택
Adapter 엔진 고유 stream을 공통 AgentEvent로 정규화
Store/UI text, tool, file, approval, done을 동일한 렌더링 계약으로 표시

Codex 전환 전 갭

현재 코드가 이미 잘하는 부분과, M6에 들어가기 전에 손봐야 할 부분을 분리했다.

03

단계별 개발 로드맵

상단 필터로 정리, MVP, 확장 단계만 따로 볼 수 있다. 순서는 C0부터 C5까지 고정하는 편이 안전하다.

04

핵심 설계 결정

Codex를 붙일 때 초기에 정해두면 이후 회귀와 UI 혼선을 줄일 수 있는 결정들이다.

05

기능 Preview 체크리스트

놓치기 쉬운 화면과 상태를 QA 관점에서 모았다. 체크 상태는 이 브라우저의 localStorage에만 저장된다.

06

테스트 전략

실 Codex 연결 전에 fake backend와 fixture를 먼저 두면 Claude 회귀 없이 M6를 진행할 수 있다.

07

주요 리스크

기능 구현보다 더 중요한 것은 provider 의미가 뒤섞이지 않게 막는 일이다.

08

Claude/Fable 작업 지시서

나중에 구현 에이전트에게 그대로 넘길 수 있는 작업 요청 초안이다. 한 번에 전부 시키지 말고 C0부터 작은 PR 단위로 끊는 것을 권장한다.
