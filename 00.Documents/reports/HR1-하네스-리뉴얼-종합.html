왜 필요한가

같은 안전 규칙이 CLAUDE.md와 AGENTS.md 두 곳에 적혀 있으면, 한쪽만 고쳐질 때 두 정본이 어긋나는 드리프트(drift)가 구조적으로 재발합니다. 실제로 문서 드리프트(H3 결함 ①)가 이미 관측됐습니다. 원인은 동기화 노력 부족이 아니라 "정본이 두 개인 구조" 자체입니다 — 그래서 코어를 엔진 중립 위치로 추출해 원인을 제거했습니다.

두 번째로, Codex는 원래 "Sol(GPT-5.6)을 직접 써보고 싶다"가 의도였는데 이식 과정에서 Claude의 운영 조직론까지 통째로 복제되어 의도와 어긋났습니다(영호 2026-07-12 풀 드라이버 전제 철회). 남겨두면 옛 결정 기반으로 사고하는 씨앗이 됩니다.

세 번째로 관측성 통증 — 훅이 echo >&2로만 말해 사용자 눈에 안 보였습니다(공식 확정: PreToolUse/PostToolUse의 stderr는 debug 로그 전용, 사용자 표시는 JSON systemMessage가 공식 수단). 안전 장치가 작동해도 안 보이면 신뢰할 수 없습니다.

GPT-5.6 Sol 적대 리뷰 = No-ship → 전건 반영

계획 단계에서 다른 AI(Sol)에게 적대적 리뷰를 받아 No-ship 판정(high 4·medium 2)이 나왔고 6건 전부 반영했습니다: 코어가 정합을 강제 못 함 → clause ID·manifest·conformance 게이트 / 계약 테스트 RED 커밋 설계 → 단일 green 원자 전환(7→6 Phase 합병) / 가변 stash@{0} 위험 → 불변 OID 앵커 / 권한 경계 제거 → 최소권한 프로필 / 로그 시크릿·동시성 → allowlist·redaction·append 직렬화 / 비가시 상태를 green 처리 → 사용자 가시 라이브 검증 의무화.

어떻게 만들었나

하네스·헌법·ADR은 영호 단독 통제 영역이라, 실행 Phase는 유지보수 창을 열어 메인 세션이 직접 수술 + 영호 감독으로 진행했습니다(선례 2026-07-11 shell-policy). Phase 문서·pin·CHANGELOG·커밋은 secretary, 제품 코드(agent-runs.ts 주석)는 main-process Worker가 맡는 분업입니다. 큰 목표를 6개 Phase로 분해하고, 각 Phase를 코어→어댑터 순으로 의존성에 맞춰 쌓았습니다.

관심사 분리를 문서에 적용"무엇이 안전인가(코어)"와 "어떻게 강제하나(어댑터)"의 분리 — 코드의 인터페이스/구현 분리와 동형.
