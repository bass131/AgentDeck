개요 — 목적과 방법

감사 목적. AgentDeck는 여러 AI 코딩 엔진을 하나의 데스크톱 IDE에서 조종하는 앱이고, 그 엔진의 대표가 곧 Claude Code입니다. 따라서 "이 앱이 IDE로서 충분한가"의 진짜 기준은 원본 GUI 클론 여부가 아니라 Claude Code가 하는 일을 GUI가 얼마나 표면화·재현하는가입니다. 이 감사는 그 격차를 체계적으로 찾습니다. FEATURE_MAP의 M1~M4·B8·B9 패리티 추적과는 보완 관계이지 대체가 아닙니다.

방법 — 4단계 워크플로. 반박 가능한 증거만 남기도록 단계를 분리했습니다: ① Claude Code 표면을 외부 문서에서 91개 기능으로 열거(external:claude-code-surface), ② 그 표면을 5개 차원으로 나눠 02.Source 코드를 차원별 실측(inventory, 총 107개 finding), ③ 실측 대비 격차 후보 58건 도출(gap), ④ 각 후보를 4개 반박 하네스(exists-already / deliberately-excluded / already-backlogged / infeasible-cli-only)로 적대 검증해 verdict 확정(verify) → 마지막으로 전부 합성(synthesize)해 마일스톤·quick win·탈락으로 정리했습니다.

규모·운영. 총 17개 Opus 에이전트(inventory 5 · external 1 · gap 5 · verify 5 · synthesize 1)가 병렬로 돌았습니다. 도중 API 오류 1건으로 일부 에이전트가 중단됐고 resume로 재실행됐습니다 — config-mcp 차원의 실측·격차가 resume 구간에서 완료됐고, 반박 검증 일부와 최종 합성이 재실행됐습니다. 같은 역할의 결과가 둘일 때는 resume 후 마지막 결과를 정본으로 채택했습니다.

synthesize:milestones · summary 전문

검증 통과 48건은 크게 세 구조적 축으로 수렴한다. (1) claude-stream.ts가 SDK system/stream 메시지를 대량 드롭하는 '이벤트 소비 계층'의 공백(sdk-events), (2) 도구 결과가 무강조 <pre>/JSON으로만 렌더되는 'IDE다운 렌더링'의 반쪽(tools-rendering — 이미 보유한 CodeViewer/AgentPanel을 도구 렌더에 재사용조차 안 함), (3) 앱이 fs read-only라 설정·권한·MCP·세션안전을 앱 밖 수기 편집에 의존하는 '제어 표면'의 부재(config-mcp + session-mgmt). 일상 임팩트×비용 비율은 M1(신호 배선)이 압도적이다 — 대부분 이미 도착한 메시지를 소비만 하면 되고 훅 투명성·루프 신뢰성 등 하네스 콕핏이라는 이 앱의 정체성과 직결된다. M2(렌더링)는 renderer 중심이라 신뢰경계 무게 없이 큰 체감을 주며 백그라운드 셸 라이브 테일이 매일의 dev/테스트 워쳐 루프를 GUI 안으로 되살린다. M3(제어·안전)는 값은 크나 read-only 아키텍처를 정면 변경하는 trust-boundary·ADR 부담이 최대라 가장 뒤·가장 게이트가 많다. 백그라운드 태스크 제어(TaskStop/Monitor)·트랜스크립트 export·엔진 버전 picker는 이미 등록된 백로그(백그라운드 태스크 수명·SDK 트랜스크립트·M5 배포)에 귀속시켜 새로 만들지 않았다.
