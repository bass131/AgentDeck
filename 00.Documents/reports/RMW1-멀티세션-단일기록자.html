🎯 무엇을 (목표)

multi-agent.json(멀티세션 단일 blob)의 lost-update 경합을 구조적으로 제거 — renderer 다중 주체의 분산 read-modify-write(RMW)를 전면 폐기하고, 병합 책임을 main 단일 기록자로 이관(ADR-031). blob 통짜 MULTI_SESSION_SAVE 채널은 소멸 — renderer 측 RMW 재발이 컴파일 타임에 불가능.

[이전 — 분산 RMW: 간극에 낀 write가 통째 소실] renderer(autosave · flush · CRUD×5 · 다중 패널) 각자: LOAD ──(IPC 왕복 간극)──► 메모리 수정 ──► SAVE(blob 통짜, last-write-wins) [이후 — 단일 기록자: 간극 자체가 소멸] renderer ──의도 명령──► main: read → merge → write (동기 원자 블록, await 0) ◄─{ok, state}── 병합 후 권위 상태 → Zustand 미러 수렴

🤔 왜 (배경·결정)

  • BF3 P05 reviewer 🟡가 발견한 선재 레이스: autosave(debounce 500ms) × 언마운트 flush × CRUD 5액션 × 다중 패널의 분산 RMW — IPC 왕복 간극에 낀 write가 last-write-wins로 소실.

  • 게이트 확정 2건(영호): ① 채널명 단일 dot camelCase(multi.cmdUpsert) ② 미지 id upsert = no-op + ok:false — 삭제된 세션을 뒤늦은 autosave가 되살리는 "stale upsert 부활" 차단.

🎓 배운 것

  1. 감지보다 제거 — CAS는 충돌을 감지하고 재시도를 남기지만, 명령 이관은 간극 자체를 소멸시킨다.
  2. run-to-completion은 공짜 락 — await 하나가 보장을 깬다. "핸들러 경로 await 0"을 완료조건으로 박은 이유.
  3. 순수 함수로 의미론 분리 — 테스트 mock이 그 함수를 그대로 import해 구현-테스트 드리프트가 구조적으로 불가능.
  4. "금지"보다 "불가능" — 채널을 없애니 위반이 컴파일 에러다. 가드레일은 타입 시스템에 박는 게 제일 싸다.
  5. stale 명령은 부활 벡터 — id 발급 주체가 유일하면 "미지 id = stale"로 거부할 수 있다. 의미론을 좁혀서 얻는 안전.
